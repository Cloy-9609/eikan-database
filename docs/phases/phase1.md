# Phase 1

## 改訂版

以下を Phase 1 の正式な解釈とする。以降の「目的」「大目標」「小目標」「完了条件」「現在地」を優先する。

## 位置づけ

Phase 1 は「基盤実装」のフェーズとする。
この段階では、手入力を前提として、アプリ全体が最低限一通り動く状態を作ることを目的とする。

Phase 0 で整理した要件・画面案・データ設計をもとに、
「登録できる」「見られる」「編集できる」「画面間を迷わず移動できる」を先に成立させる。

## 目的

- フロントエンドとバックエンドを接続し、画面操作が DB 更新に結びつく状態にする
- SQLite を使った永続化を成立させる
- 学校・選手の基本 CRUD を MVP として成立させる
- 主要な入力 UI の難所について、暫定ではなく実装可能な仕様を確定していく

## 大目標

### 1. アプリの土台を動かす

- Express サーバー起動
- SQLite 接続
- API と画面配信の接続
- DB 初期化手順の整備

### 2. 学校管理を成立させる

- 学校一覧表示
- 学校詳細表示
- 学校 CRUD API の実装
- 学校詳細から選手管理導線へつなぐ

### 3. 選手管理を成立させる

- 選手登録画面
- 選手詳細画面
- 選手編集画面
- players API の CRUD 実装

### 4. 画面遷移を成立させる

- `schools.html` から `school_detail.html`
- `school_detail.html` から `player_register.html`
- `school_detail.html` から `player_detail.html`
- `player_detail.html` から `player_edit.html`
- 保存後に適切な画面へ戻れること

### 5. 主要入力仕様を固める

- 入学年ピッカー
- 選手種別
- 都道府県選択
- ポジション選択
- スナップショット種別

## 小目標

### 1.1 サーバー・DB 初期化

- `npm start` で起動できる
- DB パスが明示される
- スキーマ未作成時に初期化できる
- 開発中に DB をリセットできる

### 1.2 schools 系

- `GET /api/schools`
- `GET /api/schools/:id`
- `POST /api/schools`
- `PATCH /api/schools/:id`
- `DELETE /api/schools/:id`
- 画面で一覧と詳細が見られる

### 1.3 players 系

- `GET /api/players`
- `GET /api/players/:id`
- `POST /api/players`
- `PUT /api/players/:id`
- 画面で登録・詳細・編集ができる

### 1.4 共通 UI / コンポーネント

- 再利用できる UI ロジックは共通化する
- 登録画面と編集画面で挙動差を極力なくす
- 仕様化した入力 UI はコンポーネントとして扱う

### 1.5 例外処理

- 不正 ID 指定時に画面が破綻しない
- API エラー時に画面上でメッセージが出る
- 必須パラメータ不足時に原因が分かる

## このフェーズでの代表成果物

- 学校一覧画面
- 学校詳細画面
- 選手登録画面
- 選手詳細画面
- 選手編集画面
- schools API
- players API
- SQLite 初期化手順
- 共通入力コンポーネント

## 完了条件

- 手入力で学校登録から選手登録、詳細表示、編集まで一通り動く
- 学校一覧から選手編集までの画面遷移が成立している
- DB 保存と再読込で状態が維持される
- 主要入力項目の仕様未確定箇所が大きく残っていない
- 入力 UI の難所について、仕様書と実装の対応関係が作られている

## このフェーズではまだ主目的にしないこと

- OCR の本格実装
- 高度な検索、絞り込み、並び替え
- 公開前提の UI 洗練
- 自動テストの大規模整備

- フロントエンドと API の接続が成立している
- SQLite 永続化が成立している
