# OCRパイプライン設計書  
**プロジェクト名**: 栄冠ナイン向け選手データ化・育成支援ツール  
**対象ゲーム**: パワフルプロ野球 2024-2025 / 将来的に 2026-2027 以降対応  
**作成目的**:  
- 「投手能力」「野手能力」「守備・起用」画面から選手情報を自動抽出する  
- 抽出結果を高校単位の管理・統計・育成支援に接続する  
- 手動修正を前提に、**高速・高信頼・保守可能**な認識基盤を設計する  

---

## 1. 背景と設計方針

本ツールは、栄冠ナインにおける以下の2軸を支えるための認識基盤を必要とする。

1. **一般選手・天才選手・転生選手のデータアーカイブ**
   - 画像認識による登録負荷軽減
   - 証拠画像付きの選手情報蓄積
   - 入学時点・育成結果・特殊能力取得履歴の統計化

2. **高校単位の育成・編成支援**
   - 3年縛りロードマップ
   - 成長履歴・差分表示
   - 起用最適化
   - ポジション穴アラート
   - 将来的な成長予測・推奨育成

そのため、OCRパイプラインは単なる文字認識機能ではなく、  
**「固定UI解析 + OCR補助 + 手動修正UX」** の三位一体として設計する。

---

## 2. 設計上の結論

### 結論
本プロジェクトの要件に対しては、**汎用OCR中心ではなく、固定UI解析中心の構成が最適**である。

### 理由
- 対象画面はゲームUIであり、**レイアウトがほぼ固定**
- 認識対象が限定的  
  - 数値
  - A〜G等のランク
  - 球速・フォーム・利き腕
  - 特殊能力名
  - ポジション・起用法
- 汎用OCR単独では誤読時の原因特定が難しく、修正UXも悪化しやすい
- 固定UI解析なら、項目ごとに
  - 切り出し
  - 認識方法
  - 辞書
  - 信頼度
  を最適化できる

### 採用方針
- **第一優先**: 固定UI解析
- **第二優先**: 項目別OCR
- **第三優先**: 手動修正
- **将来拡張**: 新作対応のための UIプロファイル追加

## 3. 要件整理

### 3.1 機能要件
OCR/認識基盤として最低限満たすべき要件は以下。

#### A. 画像認識
- 「投手能力」「野手能力」「守備・起用」の画像を取り込める
- 各画面の種別を判定できる
- 1選手あたり3枚を自動で1組にまとめられる
- 認識結果を編集可能な形で出力できる

#### B. データ化
- 選手基本情報を抽出できる
  - 名前
  - 投打
  - フォーム
  - 総合星数
  - 出身
  - 選手種別の補助情報
- 投手能力を抽出できる
- 野手能力を抽出できる
- 守備・起用情報を抽出できる
- 特殊能力を抽出できる
- 育成中に取得した特殊能力を履歴として残せる

#### C. 将来作品対応
- 2024-2025向けに検証しつつ、2026-2027以降にも対応できる拡張性を持つ

#### D. UX
- 誤認識箇所を素早く修正できる
- 2024-2025では「90%を2秒で修正可能」
- 2026-2027初期では「80%を5秒で修正可能」

---

## 4. 非機能要件

### 4.1 精度
- 重要項目で高精度であること
  - 名前
  - 数値
  - ランク
  - 特殊能力
  - ポジション
- 認識精度そのものよりも、**短時間修正可能性**を重視する

### 4.2 保守性
- 新作UIに追従しやすい構造であること
- 画面ごとの抽出ルールを個別管理できること
- 認識ロジックを差し替え可能にすること

### 4.3 可観測性
- 項目ごとに信頼度を持つ
- 元画像のどの領域から認識したか追跡できる
- 手動修正履歴を保存できる

## 5. 全体アーキテクチャ

### 5.1 パイプライン全体像

```text
[画像入力]
   ↓
[画像前処理]
   ↓
[ゲームバージョン判定 / UIプロファイル選択]
   ↓
[画面種別判定]
   ↓
[アンカー検出・位置補正]
   ↓
[項目切り出し]
   ↓
[項目別認識]
   ├─ 数値認識
   ├─ ランク認識
   ├─ 特殊能力辞書照合
   ├─ ポジション / 起用認識
   ├─ 球種 / 変化量認識
   └─ 名前 / フォーム / 備考認識
   ↓
[3枚1組 自動マージ]
   ↓
[信頼度算出]
   ↓
[手動修正UI]
   ↓
[保存導線選択]
   ├─ 新規選手作成
   │   ├─ player_series
   │   └─ players(snapshot)
   └─ 既存選手への新規時点追加
       └─ players(snapshot)
   ↓
[OCR中間保存]
   ├─ scan_session
   ├─ scan_image
   ├─ scan_bundle
   └─ recognition_result
   ↓
[統計・比較・提案ロジック]
```

### 5.2 この構成の狙い
- OCR専用の中間保存と、最終的な業務データ保存を分離する。
- 認識失敗時に、どのステップで崩れたか追跡しやすくする。
- 既存の `player_series` / `players` 設計に対し、OCRを「入力基盤」として接続できるようにする。

### 5.3 現在の導線との接続
- OCRの保存導線は、少なくとも以下の2系統を前提にする。
  - 新規選手作成導線
    - OCR結果から新規 `player_series` を作り、初回 snapshot を作成する。
  - 既存選手への新規時点追加導線
    - 既存 `player_series` に対して、指定 `snapshot_label` の snapshot を作成し、その時点を更新する。
- 特に後者では、OCRは `どの player_series / どの snapshot_label に流し込むか` を決める必要がある。
- 現在の `player_detail` / `player_edit` 導線に合わせるなら、
  `未登録時点作成 → player_edit へ遷移 → OCRアップロード / 解析結果反映 → 手動修正 → 保存`
  が最も自然な MVP 候補である。

## 8. OCRパイプライン詳細設計

### 8.1 ステップ1: アップロード受付
- 目的: 画像群をセッションとして受け取り、後続処理単位を作る。
- 処理:
  - セッションID発行
  - ファイル保存
  - メタ情報付与
- メタ情報候補:
  - 受付日時
  - ユーザーID（将来拡張。現段階では任意）
  - 高校ID（任意）
  - ゲームバージョン候補（任意）
- 出力: `scan_session`

### 8.2 ステップ2: 画像前処理
- 目的: 後続の位置合わせ・認識精度を安定化する。
- 処理:
  - リサイズ
  - 回転補正
  - ノイズ除去
  - コントラスト補正
  - UI領域のトリミング（必要なら）
  - 余白・黒帯除去
- 補足:
  - Switch / PS系キャプチャやスマホ撮影混在に備える。
  - 前処理は必ず独立モジュール化する。

### 8.3 ステップ3: ゲームバージョン判定 / UIプロファイル選択
- 目的: 対象作品に応じたUI座標・認識辞書を切り替える。
- UIプロファイルの役割:
  - レイアウト座標
  - アンカー位置
  - 色閾値
  - フォント前提
  - 特殊能力辞書
  - 項目定義
- 推奨構造:

```text
ui_profiles/
  powerpro_2024_2025/
    pitcher_ability.json
    fielder_ability.json
    defense_usage.json
    dictionaries/
  powerpro_2026_2027/
    ...
```

- バージョン判定方法:
  - ヘッダー位置・配色
  - タブ文字列位置
  - 固有アイコンのテンプレートマッチング
  - 既知UIアンカーの一致度

### 8.4 ステップ4: 画面種別判定
- 目的: 画像が `投手能力` / `野手能力` / `守備・起用` のどれかを判定する。
- 判定材料:
  - タブ領域の強調色
  - 固有ラベル文字列
  - 画面左カラムの項目構成
  - 特徴的UI要素
    - 球速 / コントロール / スタミナ
    - 弾道 / ミート / パワー
    - 起用法 / 守備位置
- 出力:
  - `screen_type`
  - `screen_type_confidence`

### 8.5 ステップ5: アンカー検出・位置補正
- 目的: 微妙な解像度差・キャプチャ位置差に対して、固定座標切り出しが成立するようにする。
- 方針:
  - 完全な固定ピクセル切り出しではなく、アンカー基準の相対座標切り出しを採用する。
- 代表アンカー候補:
  - メインウィンドウ枠
  - 顔アイコン枠
  - タブ見出し
  - `適性` / `守備位置` などのラベル
  - 左ステータス欄の罫線
- 推奨手法:
  - テンプレートマッチング
  - 色ブロック検出
  - エッジ検出 + 罫線検出
  - 相対位置補正

### 8.6 ステップ6: 項目切り出し
- 目的: 各UI項目ごとに画像領域を分離し、項目別最適認識を可能にする。
- 画面別切り出し対象:
  - 投手能力画面
    - 名前
    - 総合星数
    - 適性
    - 球速
    - コントロールランク / 値
    - スタミナランク / 値
    - 対ピンチ等の基礎特能
    - 特殊能力一覧
    - 球種
    - オリジナル変化球名
    - 投打
    - フォーム
  - 野手能力画面
    - 守備位置
    - 弾道
    - ミート
    - パワー
    - 走力
    - 肩力
    - 守備力
    - 捕球
    - 特殊能力一覧
    - 打撃成績欄（必要なら）
  - 守備・起用画面
    - メイン守備位置
    - サブポジ
    - 起用法
    - チーム方針・特性
    - 守備適性一覧

### 8.7 ステップ7: 項目別認識
- 目的: 各項目に最適な認識方式を使い分け、精度と保守性を両立する。

#### 8.7.1 数値認識
- 対象:
  - 球速
  - 各能力値
  - 星数
  - 必要なら成績欄
- 手法:
  - 数字専用OCR
  - 文字候補制約
  - 事前二値化
  - 項目ごとの桁数制約
- 値域例:
  - 球速: `100〜180`
  - 能力値: `1〜99`
  - 星数: `0〜999` 程度
- 信頼度補正:
  - 値域外は低信頼
  - 文字数過剰 / 不足は低信頼

#### 8.7.2 ランク認識
- 対象: `A〜G` 等のランク
- 手法:
  - テンプレート分類
  - 1文字OCR
  - 色 / 形状 / フォント前提の辞書照合
- 補足:
  - この項目は汎用OCRより、テンプレート照合の方が強い可能性が高い。

#### 8.7.3 特殊能力認識
- 対象:
  - 青特
  - 赤特
  - 緑特
  - 投手特能
  - 野手特能
  - 起用 / 方針ワード
- 手法:
  - 領域ごとのテキスト分割
  - 辞書照合
  - 類似文字補正
  - 正規化
    - 全角半角
    - `○` / `◯`
    - スペース揺れ
- 辞書管理:
  - ゲームバージョンごとに辞書を持つ
  - エイリアス対応を持つ
  - 将来の追加特能に追従できる形式にする
- 例:

```json
{
  "奪三振": ["奪三振", "奪三振 ", "奪 三振"],
  "キレ○": ["キレ○", "キレ◯"],
  "対ストレート○": ["対ストレート○", "対ストレート◯"]
}
```

#### 8.7.4 球種認識
- 対象:
  - ストレート
  - スイーパー
  - Hシンカー
  - フォーク
  - スローカーブ
  - その他球種
- 手法:
  - 球種表示位置ごとのOCR
  - 球種辞書照合
  - 放射状図形そのものは補助情報扱い
- 補足:
  - 球種の存在判定はテキスト優先でよい。
  - 変化量を扱う場合は別途詳細設計する。

#### 8.7.5 名前・フォーム・投打認識
- 対象:
  - 選手名
  - フォーム
  - 投打
- 手法:
  - OCR + 辞書 + 正規化
  - フォーム名は辞書照合を強く使う
  - 投打は固定パターン判定を使う
- MVPでの扱い:
  - 選手名と投打は、既存 `player_series` / `players` への正式保存候補に含める。
  - `フォーム` は現行 schema に保存先がないため、MVPでは OCR 中間結果または認識候補に留める。
  - `フォーム` を正式保存対象にする場合は、別 requirements と schema 追加が必要である。

### 8.8 ステップ8: 3枚1組自動マージ
- 目的: 投手能力・野手能力・守備起用の3画面を同一選手として統合する。
- マージキー候補:
  - 名前
  - 総合星数
  - 投打
  - フォーム
  - 顔アイコン近似
  - 同一アップロードセッション内の時系列近接
  - 高校内登録文脈
- マージ戦略:
  - 完全一致優先
  - 不完全一致時はスコアリング
  - 信頼度閾値以下は手動確認キューへ送る
- 失敗時のUX:
  - `この3枚は同一選手の可能性が高いです` と候補表示する
  - ユーザーがワンクリックで確定できるようにする

### 8.9 ステップ9: 信頼度算出
- 目的: 修正が必要な箇所をユーザーに明示する。
- 信頼度の算出要素:
  - OCRエンジンの生スコア
  - 辞書一致度
  - 値域チェック
  - 他画面との整合性
  - アンカー一致度
  - UIプロファイル一致度
- 表示方針:
  - 高信頼: 通常表示
  - 中信頼: 注意色
  - 低信頼: 強調表示 + 修正推奨

### 8.10 ステップ10: 手動修正UI
- 目的: 誤認識を短時間で修正できるようにする。
- UI要件:
  - 原画像と認識結果を並べて表示
  - 項目ごとに元切り出し画像を見られる
  - 低信頼項目を上位表示
  - 数値はテンキー入力しやすくする
  - 特殊能力は候補選択式にする
  - ランクはプルダウン選択
  - 修正後即保存できる
- 目標:
  - `2024-2025`: 90%を2秒で修正可能
  - `2026-2027` 初期: 80%を5秒で修正可能

### 8.11 ステップ11: 保存
- 目的: 認識結果とその由来をDBに保持し、後続の統計・履歴・提案に繋げる。
- 保存対象:
  - 元画像
  - 切り出し領域
  - 認識結果
  - 信頼度
  - 修正履歴
  - 最終確定データ
  - 高校との関連
- 推奨保存単位:
  - `scan_session`
  - `scan_image`
  - `scan_bundle`
  - `recognition_result`
  - `player_series`
  - `players(snapshot)`

### 8.12 保存導線の分岐
- OCR は、単に `3枚認識して保存する` だけでは不十分である。
- 最終保存前に、少なくとも以下のどちらへ流し込むかを決める必要がある。
  - A. 新規選手作成導線
    - まだ対象シリーズが存在しない場合に使う。
    - OCR結果から新規 `player_series` を作り、初回 snapshot を作成する。
  - B. 既存選手への新規時点追加導線
    - 既存 `player_series` に対して、指定 `snapshot_label` の snapshot を作成する。
    - 主に `入学時は登録済みで、1年夏大会後を追加したい` ようなケースを想定する。
- Bの導線では、現在の UX に合わせて
  `新しい時点作成 → その snapshot の player_edit へ直行 → OCR結果反映`
  を自然な基本フローとする。
- 手動修正 UI も、この `player_edit` の 1 snapshot 編集導線と接続する前提で考える。

## 9. データモデル方針（OCR関連）

### 9.1 `scan_session`
- 役割: アップロード単位
- 項目例:
  - `id`
  - `user_id`（任意 / nullable。現段階では将来拡張）
  - `school_id`（任意）
  - `created_at`

### 9.2 `scan_image`
- 役割: 画像単位
- 項目例:
  - `id`
  - `session_id`
  - `file_path`
  - `ui_profile_id`
  - `detected_screen_type`
  - `preprocessing_meta`

### 9.3 `scan_bundle`
- 役割: 3画面統合単位
- 項目例:
  - `id`
  - `pitcher_image_id`
  - `fielder_image_id`
  - `defense_image_id`
  - `merge_confidence`
  - `reviewed_by_user`

### 9.4 `recognition_result`
- 役割: 認識結果単位
- 項目例:
  - `bundle_id`
  - `field_name`
  - `raw_value`
  - `normalized_value`
  - `confidence`
  - `crop_path`
  - `corrected_value`
  - `corrected_flag`

### 9.5 既存ドメインモデルへの接続
- 選手の親概念は `player_series` を使う。
- 各時点の保存先は既存 `players` をスナップショットとして使う。
- 変化球、特殊能力、サブポジションなどの relation 系は、引き続き `players.id` 配下に保存する。
- OCR専用テーブルは、中間結果とトレーサビリティ保持に限定する。

### 9.6 MVPで正式保存しない項目
- `フォーム` は、現時点では正式保存先が未定義である。
- そのため MVP では、`scan_bundle` / `recognition_result` 側の中間結果や候補表示に留める。
- 後で必要性が確定した場合に、schema 追加とあわせて正式保存対象へ昇格させる。

## 10. 固定UI解析で要件を満たせるか

### 10.1 結論
- 条件付きで「はい」。
- 固定UI解析中心でも要件は十分満たせる。
- ただし、以下の設計条件を満たす必要がある。

### 10.2 必須条件
- UIプロファイルを作品ごとに持つこと
- 絶対座標ではなくアンカー基準の相対座標を使うこと
- 辞書照合と手動修正UIを前提にすること
- 画面種別判定と信頼度算出を組み込むこと
- 新作向けの差し替えコストを小さく設計すること

### 10.3 固定UI解析が有効な理由
- 理由1: 対象画面が定型的
  - ゲームUIは自然画像OCRと異なり、罫線・タブ・フォント・項目位置が安定している。
- 理由2: 認識対象が限定的
  - 必要なのは長文読解ではなく、数字・`A〜G`・決まった特殊能力名・決まった球種名・決まったUIラベルである。
- 理由3: UX要件に合う
  - 本プロジェクトの重要KPIは完全自動化ではなく、誤認識しても素早く直せることである。
  - 固定UI解析は `何をどう認識したか` が追いやすく、修正UIに繋げやすい。

### 10.4 固定UI解析で満たせる要件 / 条件付き要件

| 要件 | 固定UI解析での達成可否 | 備考 |
| --- | --- | --- |
| 2024-2025の高精度認識 | 可能 | 最も適している |
| 3枚1組自動マージ | 可能 | OCR結果とメタ情報で実現 |
| 手動修正の高速化 | 可能 | むしろ固定UI解析向き |
| 特殊能力認識 | 可能 | 辞書設計が重要 |
| 入学時統計用データ蓄積 | 可能 | OCR精度より保存設計が重要 |
| 育成履歴の記録 | 可能 | スナップショット設計で対応 |
| 2026-2027対応 | 条件付きで可能 | UI変更量次第 |
| 将来作品の完全自動対応 | 難しい | 新プロファイル追加が必要 |

### 10.5 固定UI解析の限界
- 限界1: UI大幅変更に弱い
  - 項目配置が大きく変わる
  - フォントや配色が変わる
  - 項目構造そのものが変わる
- 限界2: 低品質画像に弱い場合がある
  - ブレ
  - 斜め撮影
  - 反射
  - 低解像度
- 限界3: 新規項目への即時追従はできない
  - 新作で新たな特殊能力やUI要素が追加された場合、辞書や項目定義の更新が必要。

### 10.6 推奨方針
- 本プロジェクトでは `固定UI解析 + OCR補助 + 手動修正` のハイブリッドが最適。
- 優先順位:
  - 固定UI解析
  - 項目単位OCR
  - 辞書照合
  - 信頼度評価
  - 手動修正
- この方式の利点:
  - 認識速度が速い
  - 保守しやすい
  - 何が誤認識したか特定しやすい
  - 新作対応時の差分修正範囲が明確
  - UX設計と相性が良い

## 12. 例外処理とフォールバック

### 12.1 想定異常
- 画面種別が判定できない
- アンカーが見つからない
- 3枚が揃わない
- 項目の一部だけ認識できない
- 特殊能力が辞書未登録
- 画像品質が低い

### 12.2 フォールバック戦略
- 未判定画像は `手動種別選択` へ送る
- アンカー失敗時は簡易座標切り出しを行う
- 3枚不足時は部分保存可能にする
- 未知の特殊能力は `生OCR文字列` を保持する
- 低信頼時は必ず手動確認候補に送る

### 13.1 テスト観点
- 画面種別判定精度
- 項目別精度
- 特殊能力辞書一致率
- 3枚マージ精度
- 修正UXにかかる時間
- 作品差分対応性

- 正常系スクショ
- 低画質スクショ
- 類似選手名
- 多数の特殊能力パターン
- `2024-2025` 既知UI
- 将来作品仮想UI

### 13.3 主要KPI
- 以下は現時点では `必達受け入れ条件` ではなく、初期目標値 / 将来の評価指標として扱う。
- 評価データセットと計測条件が整った段階で、実測ベースへ更新する。
- 画面種別判定: `99%` 目標
- 数値認識: `95%以上` 目標
- ランク認識: `98%以上` 目標
- 特殊能力辞書一致: `90%以上` 目標
- 修正所要時間:
  - `2024-2025`: 2秒以内
  - `2026-2027` 初期: 5秒以内

## 14. 実装フェーズ提案

### 14.1 プロジェクト全体での位置づけ
- 本セクションの Phase は `OCR機能内の段階` を示すものであり、プロジェクト全体の Phase 順をそのまま意味しない。
- プロジェクト全体では、OCR は snapshot 基盤安定後の入力支援機能として位置づける。
- 少なくとも、以下がある程度安定した後に差し込むのが自然である。
  - `player_series` / `players(snapshot)` 前提の保存導線
  - `player_detail` の時点切替
  - 未登録時点作成から `player_edit` へ進む導線
- relation 系編集UIや school 側整理との前後関係は、既存 `phase2.md` を優先する。

### 14.2 Phase 1: MVP
- `2024-2025` 固定UI対応
- 3画面種別判定
- 3枚1組自動統合
- 基本ステータス認識
- `player_edit` 導線に載せる前提の手動修正UI
- DB保存

### 14.3 Phase 2: 実用化
- 特殊能力辞書強化
- `player_detail` / `player_edit` からの自然な OCR 反映導線
- 高校管理画面からの直接登録
- 信頼度表示
- 切り出し画像プレビュー
- 修正ログ保存

### 14.4 Phase 3: 将来拡張
- `2026-2027` UIプロファイル追加
- 一括スキャン強化
- 類似画像マッチング
- 未知UI検知
- 統計基盤との密結合

### 14.5 初期MVPから外す候補
- `フォーム` の正式保存
- 顔アイコン近似による高度マージ
- 動画 / 連続キャプチャ対応
- 過度に厳しい KPI 達成保証

## 15. 最終判断

### 15.1 本プロジェクトにおける最適解
- 本ツールのOCR基盤は、固定UI解析を主軸に据えて問題ない。
- むしろ本要件に対しては、固定UI解析の方が適している。

### 15.2 ただし前提
- `固定UIだから将来作品にもそのまま無改修で通る` という意味ではない。
- 正確には、以下が妥当である。
  - 固定UI解析を、UIプロファイル差し替え可能な構成で実装すれば、将来作品にも十分現実的に追従できる。

### 15.3 推奨する最終方針
- `2024-2025` で固定UI解析を完成させる
- その際にプロファイル差し替え構造を最初から入れる
- `2026-2027` 発売後にUI差分だけを更新して追従する
- 精度不足は手動修正UIと信頼度表示で吸収する

## 16. 補足メモ

### 16.1 OCRパイプラインの位置づけ
- OCRパイプラインは本ツール全体において入口であり、この基盤の質が次の機能の質を決める。

### 16.2 この基盤が影響するもの
- 選手アーカイブの信頼性
- 入学時点統計の精度
- 成長履歴の正確性
- 起用最適化の妥当性
- 成長予測の学習データ品質

### 16.3 本設計で優先すること
- 認識精度の最大化よりも、認識結果の追跡可能性・修正容易性・保守容易性を優先する。
- これが、栄冠ナイン向け実運用ツールとして最も現実的で強い設計である。
