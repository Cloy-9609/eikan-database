# 守備位置図UI（player_detail 用）仕様メモ

## 目的
選手詳細画面において、守備・起用情報を文章や一覧だけでなく、
野球の守備位置図として視覚的に把握できるようにする。

主な狙いは以下の通り。

- 見た瞬間に「どこを守れるか」を理解しやすくする
- メインポジションとコンバート候補の適性差を把握しやすくする
- `player_detail` の守備・起用セクションを、一覧や通常フォームとは異なる専用 UI として強化する

## 導入タイミング
- `player_detail` における各能力の表示が成立していること
- `player_edit` における各能力の編集・保存が成立していること
- その後、Phase 3 に入る前の Phase 2 後半で着手する

## 想定画面
- 主対象: `frontend/pages/player_detail.html`
- 将来的には `player_edit.html` でも補助表示として流用可能

## 想定UI
- 背景に簡略化した野球場を表示する
  - 外野
  - 内野
  - 塁線
  - ホームベース
- その上に各守備位置スロットを重ねる
- まずは読み取り用 UI を優先し、「守れる場所の可視化」を主目的とする
- 一覧画面や通常フォームとは切り離し、守備・起用専用コンポーネントとして扱う

## 対象ポジション
- 中堅
- 左翼
- 右翼
- 二塁
- 遊撃
- 三塁
- 一塁
- 投手
- 捕手

## 配置ルール
- 中堅: 外野中央上部
- 左翼: 外野左側
- 右翼: 外野右側
- 二塁: 内野中央右寄り
- 遊撃: 内野中央左寄り
- 三塁: 内野左側
- 一塁: 内野右側
- 投手: 内野中央下寄り
- 捕手: ホームベース後方中央

### 特記事項
- 二塁手と遊撃手は、同じ高さで左右対称に置く
- 表示対象は、メインポジションと現在のデータで表現可能なコンバート先・サブポジションとする
- メインポジションがある場合は強調表示する
- コンバート候補やサブポジ適性も表示できる構造にしておく

## 各スロットの表示候補
- 守備位置名
- 適性ランク
- 守備力数値
- メインポジション表示
- 守備不可時の表示ルール
  - 薄表示
  - 非表示
  - `適性なし` 表示

## 想定データ
初期段階で参照したいデータ候補は以下。

- `main_position`
- `player_sub_positions`
- 野手守備関連の数値
  - `fielding`
  - `catching`
  - 必要に応じて肩力など
- 将来的な守備適性評価用の補助値

### データ面の考え方
- 表示ロジックと描画 UI は分離する
- 現在のデータ構造で表現可能な守備適性を先に表示し、コンバート後守備力の詳細反映はデータモデル整理後に拡張する
- 将来的に以下を区別できるようにする
  - メインポジション
  - サブポジション
  - コンバート時の適性
- 初期導入時点では、現在のデータモデルで安全に表現できる範囲に絞る

## 実装前提
- `player_detail` の各能力表示が概ね固まっていること
- `player_edit` の各能力編集・保存が成立していること
- 守備・起用セクションの配置や見出し階層が整理されていること
- メインポジション、サブポジション、適性ランクの扱いが文書化されていること

## 懸念点
- 現在のデータモデルで「コンバート後の適性」をどこまで自然に表現できるか
- UI を先行しすぎると、あとでデータ構造とズレる可能性がある
- 守備位置名の正式値と enum 管理の整合が必要
- レスポンシブ時に位置図とスロットの可読性をどう保つか検討が必要

## 技術方針候補

### 案1: SVG ベース
- 拡大縮小に強い
- 守備位置座標を扱いやすい
- 背景グラウンドを独自デザイン化しやすい

### 案2: CSS 描画 + absolute positioning
- 既存の素の HTML/CSS/JS 構成に馴染みやすい
- 実装コストは比較的低い
- スロット配置や余白調整はしやすい

## 推奨方針
- 背景グラウンド: SVG または簡略な CSS 描画
- 守備位置スロット: absolute positioning
- 座標定義: JS オブジェクトで管理
- 描画処理: 表示ロジックから分離

この形であれば、専用 UI としての見た目を作りつつ、
既存の素の HTML/CSS/JS 構成とも馴染ませやすい。

## 先行条件
- `player_detail` の表示完成
- `player_edit` の編集・保存完成
- 守備適性の表示ルール整理
- `player_detail` の section 構成整理

## 備考
- 一覧画面や通常フォームへの流用は前提にしない
- まずは `player_detail` の読み取り体験改善を優先する
- 将来的に起用支援や配置 UI に発展する余地はあるが、初期導入ではそこまで広げない
- 「選手を守備位置図に配置できる機能」は別機能として扱い、本メモの対象外とする
