# 選手スナップショット時系列管理 仕様メモ

## 目的
- 現行の `snapshot_label` 単独管理から脱却し、同一選手に複数時点の状態を持たせる。
- 既存 `players` テーブルを「各時点のスナップショット」として活かしつつ、同一選手を束ねる親概念を追加する。
- `player_detail` 上で、各時点の閲覧・追加・編集を自然に行えるようにする。

## 採用方針
- 案Aを採用する。
- 新たに親テーブル `player_series` を追加し、既存 `players` は各時点スナップショットの子テーブルとして再利用する。
- `players` には `player_series_id` を追加し、1人の選手に複数スナップショットを紐づける。
- 変化球、特殊能力、サブポジションなどの relation 系テーブルは、引き続き `players.id` に紐づける。

## 正式な時点定義
表示順・時間順は以下で固定する。

| 順序 | 内部キー | 表示名 |
| --- | --- | --- |
| 1 | `entrance` | 入学時 |
| 2 | `y1_summer` | 1年夏大会後 |
| 3 | `y1_autumn` | 1年秋大会後 |
| 4 | `y1_spring` | 1年春大会後 |
| 5 | `y2_summer` | 2年夏大会後 |
| 6 | `y2_autumn` | 2年秋大会後 |
| 7 | `y2_spring` | 2年春大会後 |
| 8 | `y3_summer` | 3年夏大会後 |
| 9 | `graduation` | 卒業時 |

### 時点定義ルール
- 表示名と内部キーは分離する。
- 並び順は共通定数で管理し、DBに自由入力させない。
- 1つの `player_series` に対して、同じ内部キーのスナップショットは1件までとする。
- `snapshot_label` は子スナップショットの時点キーとして扱う。初期実装では既存カラム名を活かし、必要なら後続で `snapshot_key` へ改名を検討する。

## 親子構造

### 採用名称
- 親概念の名称は `player_series` を採用する。

### `player_series` を採用する理由
- 同一選手の「時系列のまとまり」を自然に表現できる。
- `player_groups` よりも責務が明確で、単なる雑多な束ねに見えにくい。
- `player_identity` よりも実装名として扱いやすく、認証や実在人物識別の意味と混同しにくい。

- `school_id`
- `name`
- `prefecture`
- `player_type`
- `player_type_note`
- `admission_year`
- `throwing_hand`
- `batting_hand`
- `common_memo`
- `created_at`
- `updated_at`

### 子 `players` に持たせる情報
- `id`
- `player_series_id`
- `snapshot_label`
- `main_position`

### relation 系で子に紐づける情報
- `player_pitch_types`

### 設計上の補足
- `school_id`、名前、出身都道府県、選手種別、投打、入学年は親の共通情報として扱う。
- `main_position` はコンバートを表現したいため、子スナップショット側で管理する。
- `grade` は時点キーから導出できるため、最終形では「表示用導出値」扱いを推奨する。互換維持のため当面DB列を残す場合でも、自由編集値ではなく時点キー整合を優先する。
- `snapshot_label` はスナップショット識別子の一部であるため、通常運用では作成後の変更を原則不可とする。修正が必要な場合は移行用または管理用導線で扱う。

## `player_detail` の表示構造

### 基本構成
- 上段: `player_series` の共通情報
- その直下: 時点ボタン群
- 下段: 現在選択中スナップショットの能力、守備・起用、特殊能力、変化球

### 初期表示ルール
- `player_detail` 初期表示時は、登録済み時点のうち、正式定義一覧で最も下にあるものを表示する。
- つまり、時間的に最も新しい登録済みスナップショットをデフォルト表示する。
- URL で時点指定があり、その時点が登録済みならそちらを優先する。

## 時点ボタン UI 仕様

### 共通ルール
- ボタンは正式な9時点を固定順で並べる。
- 各ボタンは、登録済みか未登録かを見た目で区別できるようにする。
- 選択中の時点は、登録済みボタンの中でさらにアクティブ状態を持つ。

### 登録済みボタン
- 色付きボタンとする。
- 押すと、その時点の詳細表示へ切り替える。
- URL や local state には「現在選択中の時点キー」を保持できるようにする。

### 未登録ボタン
- 空ボタンまたはアウトラインボタンとする。
- 押した瞬間に新規登録画面へ飛ばさない。
- ボタン押下後に、軽量な確認表示を出す。
- 文言は `作成しますか？` を基本とする。
- 確認表示内に `作成する` ボタンを置き、そこから新規登録画面または新規スナップショット作成導線へ遷移する。

### 確認表示の推奨方式
- 初期採用は、ボタン近傍に出る小さな確認ポップオーバーを推奨する。
- 理由は、ツールチップより操作要素を置きやすく、ドロップダウンより意味が明確だからである。
- 実装上は、将来ツールチップ風や小テロップ風へ差し替えられる構造にする。

### UX要件
- 確認表示はON/OFFを切り替え可能にする。
- 初期段階では local state または `sessionStorage` 程度で保持すればよい。
- 「確認を毎回出す」「このセッションでは省略する」のような軽い設定から始める。
- UIの凝った再現より、登録状況の分かりやすさを優先する。

## 新規スナップショット作成ルール

### 基本ルール
- 新しい時点を作成するときは、その時点より前で最も近い登録済みスナップショットを探す。
- 見つかった場合は、その内容を初期値としてコピーする。
- 以前の時点が1つもなければ、親情報または入学時相当の初期値を使う。

### 初期値ソースの優先順位
1. 直前の登録済みスナップショット
2. `player_series` の共通情報
3. システム既定値または空初期値

### 項目ごとの考え方
- `main_position`: 基本は直前スナップショットを継承する。コンバート後に新規時点を作る場合は、変更後ポジションが次の初期値になる。
- `throwing_hand` / `batting_hand`: 親の共通情報として扱い、新規作成時は親情報をそのまま適用する。互換都合で子にも値を持たせる場合は、親値と一致させる。
- 能力値: 直前スナップショットをコピーし、その時点に合わせて編集する。
- `total_stars`: 直前スナップショットをコピーし、必要に応じて更新する。
- `evidence_image_path`: 原則コピーしない。新時点ごとに別証跡を持つ想定とし、空で開始してよい。
- `snapshot_memo`: 直前をそのまま複写せず、空初期値を推奨する。必要なら「前回メモをコピー」の補助操作を別途用意する。

## 特殊能力・変化球の扱い方針

### 推奨方針
- 特殊能力・変化球は、各スナップショットごとに「その時点での完成形一覧」を保持する。
- 新規スナップショット作成時には、直前時点の内容を自動コピーして初期値にする。
- その後、追加・変更・削除を通常編集で反映する。

### この方針のメリット
- 読み取り時に差分計算が不要で、`player_detail` 表示が単純になる。
- UI が「その時点で持っている一覧を編集する」形になり、理解しやすい。
- relation 系テーブルの既存構造を大きく崩さずに済む。
- 1選手あたり最大9時点なので、一覧複製によるデータ重複コストは許容しやすい。

### 注意点
- データは増えるが、9時点上限を前提に受け入れる。
- 「後から消えた能力」の解釈は、差分履歴ではなくスナップショット比較で行う。
- 直前コピー後に編集せず保存すると、意図せず同一内容の時点が続く可能性があるため、作成直後に変更点確認UIがあると望ましい。

## 一覧画面・導線への影響
- `school_detail` の所属選手一覧は、最終的に `players` 直列ではなく `player_series` 単位表示へ移行する。
- 一覧カードには、共通情報と最新登録スナップショットの要約を出す。
- `player_detail` の導線は、将来的に `player_series_id` 基準へ寄せる。
- 互換維持のため、当面は `player_id` 直指定から `player_series_id + snapshot_label` を解決する移行層を持ってよい。

## 旧値・移行方針

### 旧 `snapshot_label` 値の扱い
- `post_tournament` は正式運用値から外す。
- `after_1st_summer` など旧命名は、正式値 `y1_summer` などへ寄せる。
- 旧値を混在させると比較・並び順・一意制約が複雑になるため、最終的には正式9値へ統一する。

### 移行上の注意
- `post_tournament` はどの大会後かを一意に判断できないため、自動移行だけでは不十分である。
- 既存データに `post_tournament` が残っている場合は、確認用の移行UIまたは手動マッピング手順を用意する。
- `player_series` 作成時は、まず既存 `players` 1件に対して `player_series` 1件を仮生成し、後から同一選手スナップショットを統合できるようにする。

## 想定変更範囲

### DB
- `player_series` テーブル追加
- `players.player_series_id` 追加
- `player_series_id + snapshot_label` の一意制約追加
- `snapshot_memo`、`common_memo` など新規メモ欄追加の要否整理
- `snapshot_label` の許可値を正式9値へ統一
- `grade` を導出値扱いに寄せるか、互換列として残すかを決定

### API
- `GET /api/players/:id` 相当の返却を、単一 snapshot だけでなく「親 + スナップショット一覧 + 現在表示 snapshot」を返せる形へ拡張
- `POST /api/players` を「新規選手作成」と「既存選手への新規スナップショット追加」で責務整理
- 新規スナップショット作成時に、直前 snapshot のコピー初期化をサーバー側で扱えるようにする
- 親共通情報編集と snapshot 編集の責務分離を進める

### フロントエンド
- `player_detail.js`: 時点ボタン群、最新時点初期表示、未登録時点の確認表示、表示切替
- `player_register.js`: 新規選手作成モードと既存選手への新規時点追加モード
- `player_edit.js`: 1スナップショット編集前提への整理
- `school_detail.js`: 一覧を最終的に `player_series` 単位へ寄せる
- 共通定数: 正式9時点キー、ラベル、順序を frontend/backend で共有または同期する

## 実装段階

### Phase A: 設計固定
- 正式な9時点キーを確定する。
- `player_series` の責務と子 `players` の責務を確定する。
- `player_detail` の時点ボタンUI仕様と未登録時導線を確定する。
- relation 系編集UIより先に、この時点設計を優先する。

### Phase B: DB / API 土台
- `player_series` テーブル追加
- `players.player_series_id` 追加
- `player_series_id + snapshot_label` の一意制約追加
- 既存 `players` 取得APIに、同一 `player_series` のスナップショット一覧を返す拡張を入れる
- 新規スナップショット作成APIに、直前コピーロジックを入れる

### Phase C: `player_detail` 時点切替
- `player_detail` を `player_series` 中心の読み取り画面へ切り替える
- 時点ボタン群を表示する
- 最新登録済みスナップショットを初期表示する
- 未登録ボタンから確認表示を経て作成導線へ進める

### Phase D: 作成・編集導線
- `player_register` を「新規選手作成」と「既存選手への新規スナップショット追加」の両対応にする
- `player_edit` は1スナップショット編集を基本責務に寄せる
- 親共通情報の編集は `player_series` 編集として分離する

### Phase E: relation 系編集と派生UI
- 特殊能力、変化球、サブポジション編集をスナップショット前提で実装する
- その後に守備位置図などの派生UIへ進む

## 今すぐ実装を始めるなら
1. `player_series` と正式9時点キーを requirements / API 契約で先に固定する。
2. `player_detail` の返却データを「親 + スナップショット一覧 + 現在表示スナップショット」に整理する。
3. `player_register` に「既存選手へ新規時点を追加する」モードを追加する。
4. 旧 `post_tournament` の扱いを決める移行方針を先に潰す。
