# 学校一覧 advanced 検索・ソート 将来設計

## 目的
- `schools.html` の basic 検索・ソートを先に安定させた上で、次段の advanced 検索・ソートを安全に拡張するための論点を整理する。
- 今回は実装しない機能を、API / SQL / UI の観点で将来着手しやすい形で残す。

## 今回見送る対象

### 1. 都道府県独自順ソート
- 対象:
  - 北海道から順
  - 沖縄から順
- 今回見送る理由:
  - 文字列順ではなく独自マスタ順が必要になる。
  - backend / frontend のどちらを source of truth にするかを先に決める必要がある。
  - basic ソートより設計判断が一段重い。
- 将来設計案:
  - backend 側に都道府県順序マスタを持ち、`ORDER BY CASE` で順序を固定する。
  - 海外候補を都道府県列と同一ソート軸に含めるか、別グループで後ろへ寄せるかを仕様化する。
  - query 例:
    - `sort_by=prefecture_custom`
    - `sort_order=asc|desc`
- 追加検討点:
  - API の許可値追加方法
  - SQL の順序定義の持ち場所
  - UI 表示名を「都道府県順」「都道府県逆順」などにするか

### 2. 所属選手人数順ソート
- 対象:
  - 昇順
  - 降順
- 今回見送る理由:
  - 学校一覧取得時に player count 集計が必要になる。
  - SQL と response shape が basic 一覧より重くなる。
  - 一覧性能と設計の影響範囲が広がる。
- 将来設計案:
  - schools 一覧取得で `LEFT JOIN players` と `COUNT(players.id)` を使い、学校ごとの所属選手数を集計する。
  - アーカイブ済み学校やアーカイブ済み学校所属選手をどう集計から除外するかを明確化する。
  - response に `player_count` を含めるか、ソート専用の内部列として扱うかを決める。
  - query 例:
    - `sort_by=player_count`
    - `sort_order=asc|desc`
- 追加検討点:
  - 一覧レスポンスへ `player_count` を公開するか
  - 集計付き一覧と通常一覧を同一 API で扱うか
  - 将来の pagination 導入時の整合性

### 3. 所属選手名での学校検索
- 対象:
  - 所属選手名の部分一致で学校を逆引きする検索
- 今回見送る理由:
  - schools と players の join が必要になる。
  - 学校名検索とは別系統の検索仕様になる。
  - UI でも「学校名検索」と混ぜると意図が分かりにくくなる。
- 将来設計案:
  - `GET /api/schools` に直接載せるか、専用 query を追加するかを検討する。
  - 例:
    - `player_name`
  - SQL は `EXISTS` または `JOIN` を使い、学校単位で重複を排除する。
- 追加検討点:
  - 学校名検索との優先順位と AND / OR 条件設計
  - アーカイブ済み学校所属選手の検索対象可否
  - UI を同一パネルに置くか、advanced 条件へ分離するか

## Phase 2 後半以降で扱う理由
- まずは学校管理の入口画面として、basic 検索・basic ソート・作成導線整理を安定させる価値が高い。
- advanced 機能は API 契約、SQL 負荷、UI の複雑度に追加判断が必要で、今回の安全な範囲を超える。
- 将来は pagination、件数増加、一覧集計、複合検索 UI と合わせて設計した方が一貫性を保ちやすい。

## 今回との差分整理
- 今回実装するもの:
  - 学校名部分一致検索
  - 都道府県完全一致検索
  - プレイ方針完全一致検索
  - 学校名順、開始年度順、更新日時順のソート
- 今回実装しないもの:
  - 都道府県独自順ソート
  - 所属選手人数順ソート
  - 所属選手名での学校逆引き検索
