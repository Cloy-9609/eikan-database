# 学校管理 将来設計メモ

## 目的
Phase 1 では学校 CRUD と学校情報の整理を優先し、将来拡張が必要な学校管理 UI / 年度更新機能はここに分離して記録する。

## school_detail 右上アクションメニュー構想
- `school_detail.html` の右上に 3 点メニューまたはハンバーガーメニューを配置する。
- 初期候補アクション:
  - 学校情報を編集
  - 学校を削除
  - 年度を更新
- Phase 1 では常時表示の編集フォームを採用し、メニュー化は見送る。
- 将来は summary 画面を主、編集を副操作に寄せることで、確認画面としての視認性をさらに高める。

## current_year 独立更新の考え方
- Phase 1 では `current_year` は `start_year` と同期し、独立操作は持たせない。
- 将来の年度更新機能では、`current_year` を `start_year` から独立して +1 年ずつ進める。
- 想定 API / 操作例:
  - `POST /api/schools/:id/advance-year`
  - school detail のアクションメニューから「年度を更新」
- 年度更新では `start_year` は固定し、`current_year` のみ変化させる前提とする。

## 年度更新時の注意点
- `current_year` を更新した時点で、学校 summary の経過年数表示も連動更新する。
- 年度更新後に player 側で何を再計算するかは Phase 2 以降で別途設計する。
- 年度更新履歴が必要か、監査ログを残すかは将来判断とする。
- 誤操作防止のため、独立更新機能には確認ダイアログまたは undo 導線を検討する。

## Phase 2 以降で扱う理由
- Phase 1 の主眼は学校 CRUD の完成度向上と基本情報の整理である。
- 右上アクションメニューは UX 改善として有効だが、必須導線ではない。
- `current_year` の独立更新は player / 成績 / 表示への影響範囲が広く、Phase 2 以降で検索性や運用導線と合わせて設計した方が安全である。
