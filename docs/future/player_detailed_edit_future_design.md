# 選手詳細編集 将来設計メモ

## 目的
Phase 1 では選手の基本編集と「未編集項目を保持する」安全性を確保し、詳細能力編集 UI は将来機能として分離する。

## 未実装の詳細能力編集 UI
- 投手能力:
  - 球速
  - コントロール
  - スタミナ
- 野手能力:
  - 弾道
  - ミート
  - パワー
  - 走力
  - 肩力
  - 守備
  - 捕球
- 変化球一覧:
  - 球種名
  - 変化量
  - オリジナルフラグ
  - オリジナル名
- 特殊能力一覧:
  - 能力名
  - カテゴリ
  - ランク値
- サブポジ一覧:
  - ポジション名
  - 適性値

## 保存単位の検討
- 候補 1: 一括保存
  - `player_edit.html` に全セクションを載せ、1 回の保存で player 本体と関連テーブルをまとめて更新する。
- 候補 2: section 単位保存
  - 基本情報、投手能力、野手能力、変化球、特殊能力、サブポジを分けて保存する。
- Phase 2 以降では、UI の複雑さと更新競合を踏まえてどちらを採るか決定する。

## 現行 service merge との関係
- 現在の `playerService.updatePlayer()` は、編集画面で未送信の項目を既存値で保持する。
- 詳細能力編集 UI を追加する場合も、この merge 層を維持して「編集していない section は壊さない」前提を保つ。
- 一括保存にする場合は、全 section の payload をまとめて検証する。
- section 単位保存にする場合は、partial update を受けても merge 層で関連データを安全に保持できる設計が必要になる。

## Phase 2 以降で扱う理由
- Phase 1 では学校管理と基本 CRUD の完成度を優先する。
- 詳細能力編集 UI は入力項目数が多く、設計と検証コストが高い。
- 検索・一覧性・OCR 連携などの Phase 2 機能と合わせて、詳細データの入力導線を再設計した方が全体整合が取りやすい。
