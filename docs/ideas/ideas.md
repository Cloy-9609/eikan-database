## ・ 学年一括進級機能

種別: 機能案  
優先度: 中  
状態: 未検討  
採用候補時期: Phase2以降  

### 目的
高校管理で年度更新をまとめて処理したい

### 仮仕様
- 高校詳細画面でボタン
- 全選手の grade を +1
- 3年は卒業扱い

### 懸念
- 卒業生の扱い
- 成績や履歴の保存方法
- 卒業タイミングのズレ（大会途中など）

### 原案アイデア
* 高校詳細画面でボタン
* 全選手のgradeを+1
* 3年は卒業扱い
* 卒業生は削除せず、現役生の下に卒業生一覧を表示し、移す
* 卒業済みフラグを立て、卒業後の就職先を記載
* 成績、能力は残す。現役生と同様に選手データの閲覧を可能なまま（卒業生と分かるUI、色分けは有りか）

## ・ 転生・天才選手自動判定

種別: 機能案(画像認識機能の一部)  
優先度: 低  
状態: 未検討  
採用候補時期: 画像認識システム完成後、ツール全体の完成後でも可  

### 目的
画像認識による選手取り込みの時点で転生選手・天才選手の区別をつけられるようにしたい  

### 仮仕様(自動判定寄り)
- 天才か否かで{-,〇}を表示する項目を追加 or 選手ステータスに「性格」を追加
- 転生の有無で{-,〇}を表示する項目を追加
- 上記2点を画像認識で判別
### 仮仕様(判定補助寄り)
- OCRで候補を取得
- DBと照合
- 「転生候補」「天才候補」として提示
- 最終判断はユーザー

### 懸念
- 転生且つ天才選手の可能性が0ではない
- テーブルに項目を追加する必要が生じる可能性
- 性格は読み込む3枚の画像に記載がないため、天才でない性格の時、手動入力が必須となる  
- 性格は育成中に手動で変えることが可能なため、or以降の機能を実装せず、性格はデータとして取得しない方向性も有りか

### 原案アイデア
* OCRで名前一致
* DBと照合

### 選手詳細の守備・起用を守備位置図で可視化する機能

種別: 機能案  
優先度: 中〜高  
状態: 仕様整理中  
採用候補時期: Phase2後半（`player_detail` の各能力表示・編集・保存成立後、Phase3前）  

### 目的
`player_detail.html` において、守備・起用情報を文章や一覧だけでなく野球の守備位置図として視覚化し、
「どこを守れるか」「メインポジションとコンバート先でどの程度差があるか」を一目で理解できるようにする。

※ これは `player_detail` の読み取り用可視化 UI の案であり、
「選手を守備位置図に配置できる機能」とは別機能として扱う。

### 仮仕様
- 対象画面は `player_detail.html` とし、一覧画面や通常フォームとは別の専用 UI として扱う。
- 背景に簡略化した野球場を表示し、その上に各守備位置スロットを重ねる。
- 対象ポジションは `中堅 / 左翼 / 右翼 / 二塁 / 遊撃 / 三塁 / 一塁 / 投手 / 捕手` とする。
- 各スロットには、守備位置名、適性ランク、守備力数値、メインポジションかどうかを表示できるようにする。
- 二塁手と遊撃手は、内野中央で同じ高さ・左右対称に置く。
- 将来的に `main_position / sub_positions / コンバート時の適性` を反映できるよう、表示ロジックと見た目を分離する。
- 初期段階では読み取り用 UI を優先し、編集・ドラッグ操作は後段で再検討する。

### 実装前提
- `player_detail` で各能力の表示が成立していること。
- `player_edit` で各能力の編集・保存が成立していること。
- メインポジション、サブポジション、守備適性値の扱いが整理されていること。
- 守備・起用セクションの表示順や section 構成が概ね固まっていること。

### 懸念
- 現在のデータモデルで「コンバート時の適性」をどこまで表現できるか未確定。
- UI 先行で作ると、後から能力項目や適性計算ルールとズレる可能性がある。
- 位置図は専用 UI なので、汎用フォーム部品とは別の保守方針が必要。
- SVG / CSS 描画 / absolute positioning のどれを主軸にするかは、レスポンシブ性と保守性を比較して決める必要がある。

### 原案アイデア
* 選手詳細で守備位置図を表示し、適性を視覚化する
* メインポジションを強調し、サブポジ・コンバート候補との差分を把握しやすくする
* 将来的な起用支援 UI の前段として、まずは読み取り用コンポーネントから導入する

### 選手を守備位置図に配置できる機能

種別: 機能案  
優先度: 高  
状態: 未検討  
採用候補時期: Phase2完了後 または Phase3以降  

### 目的
高校内の選手を実際に図上で守備位置につかせ、
人員不足や、ポジションごとに必要ステータスの確認などをしやすくする。

※ これは `player_detail` の読み取り用守備位置図とは別機能とする。

### 仮仕様
- `school_detail.html` から本機能へ遷移する導線を追加する
- 画面全体に守備位置図、画面右に選手一覧を表示する
- 画面右の選手一覧では、選手名と守備位置（コンバート含む）が分かりやすく表示されている
- 選手一覧は、表示切替や展開により全員の能力を参照できる
- 各守備位置アイコンから選手を選択できる
- 将来的には、右側の選手一覧からドラッグアンドドロップで配置する案も検討する
- 各守備位置にはスターティングメンバーだけでなく、代打、代走、守備固め要員などを配置する拡張余地を残す

### 懸念
- 配置 UI とデータ保存単位をどう切り分けるか未整理
- 守備位置図の読み取り用 UI と責務が混ざると複雑化しやすい
- ドラッグアンドドロップを採るか、クリック選択式を採るかで実装コストが変わる
- 起用履歴や保存方式まで含めると、別フェーズ扱いが必要になる可能性がある

### 原案アイデア
* 選手の守備位置図への配置
* 右側の選手一覧から配置する
* スタメンだけでなく、控え要員配置にも拡張する

### 選手スナップショット時系列管理の再設計

種別: 機能 / データ設計  
優先度: 高  
状態: 方針確定（案A採用）  
採用候補時期: Phase2後半〜Phase3前  

### 目的
`snapshot_label` を単なる基本情報の1項目として扱う状態から脱却し、
1人の選手に対して複数時点のスナップショットを自然に保持・閲覧・追加できるようにする。

### 仮仕様
- 既存 `players` テーブルは各時点のスナップショットとして活かす
- 同一選手を束ねる親概念として `player_series` を追加する
- `player_detail` は親の共通情報 + 時点ボタン群 + 選択中スナップショット表示の構成にする
- 登録済み時点ボタンは色付き、未登録時点は空ボタンとして区別する
- 未登録時点押下時は即遷移ではなく、小さな確認表示を出してから新規登録画面へ進める
- 新規時点作成時は、直前の登録済みスナップショットを初期値コピーする
- 特殊能力・変化球は各スナップショットごとの完成形を保持し、新規作成時に直前コピーする

### 懸念
- 旧 `post_tournament` 値は正式時点へ自動で割り振れないため、移行方針が必要
- `school_detail` 一覧や `player_detail` ルーティングも、最終的には親単位へ寄せる必要がある
- relation 系編集 UI を先に作ると、後で親子構造へ合わせ直すコストが増える

### 原案アイデア
* `player_series` + `players(snapshot)` の親子構造にする
* `player_detail` で時点切替を行う
* 直前スナップショットを初期値コピーして新時点を作成する
* relation 系編集はこの設計確定後に進める
