# 目的

`frontend/css/player.css` について、今後の UI 調整・ゲーム風デザイン適用をしやすくするため、現在の見た目を大きく壊さない範囲で
**UIトークン（色・余白・角丸・枠線・影・文字サイズなど）の整理** を行いたいです。

今回は見た目を大きく変えることよりも、
**CSS のデザイン刷新ではなく、変数化と責務整理、再利用性・一貫性・保守性を上げること** を目的にします。

---

# 前提

* `player_register` / `player_edit` / `player_detail` のページ骨格はある程度共通化済み
* `player.css` にはすでに `:root` の変数が一部ある
* ただし、まだ色・余白・角丸・影などがハードコードで散っている
* 年ピッカー、フォーム、detail カード、ボタンなどが同一ファイル内にある
* 実ブラウザ確認では大きな表示崩れは出ていない
* 今後「野球ゲーム風UI」に寄せる予定だが、完全再現ではなく独自UIとして整理したい
* 以前まとめた安全ガイドの方針に従い、**パワプロそのものではなく、ゲームUIに寄せやすい土台** を作りたい

---

# 制約条件

* 機能の挙動は変えないこと
* API仕様変更は禁止
* DBスキーマ変更は禁止
* HTML構造やJSは、必要がなければ変更しないこと
* 見た目は大きく壊さないこと
* 変更は基本的に `player.css` 中心で行うこと
* 差分は最小限にすること
* 「トークン整理」が主目的であり、全面デザイン刷新ではないこと
* 既存の UI を壊さず、将来調整しやすい形にすること
* 「パワプロ完全再現」ではなく「野球ゲーム風UIとして独自に育てる」前提を守ること
* 既存の実ブラウザ確認で問題なかったレイアウトを崩さないこと

---

# 作成対象

* `frontend/css/player.css`

必要なら最小限:

* `frontend/pages/player_register.html`
* `frontend/pages/player_edit.html`
* `frontend/pages/player_detail.html`

ただし、今回は原則 CSS のみで進めてください。

# 実装内容

## 1. UIトークンの整理

`:root` にある変数を拡張し、少なくとも以下の種類をトークンとして整理してください。

### 色

* 背景色
* 面の色
* 枠線色
* 強い枠線色
* テキスト色
* 補助テキスト色
* アクセント色
* エラー色
* 成功色

例:

* `--player-bg`
* `--player-surface`
* `--player-surface-solid`
* `--player-border`
* `--player-border-strong`
* `--player-text`
* `--player-text-muted`
* `--player-accent`
* `--player-accent-strong`
* `--player-success-*`
* `--player-error-*`

### サイズ

* 入力欄高さ
* ボタン高さ
* 年ピッカー数字窓高さ
* 小文字 / 通常 / 見出しサイズ


### 余白

* 小 / 中 / 大 の spacing
* カード内 padding
* フォーム要素の gap

例:

* `--player-radius-sm`
* `--player-radius-md`
* `--player-radius-lg`
* `--player-space-xs`
* `--player-space-sm`
* `--player-space-md`
* `--player-space-lg`
* `--player-space-xl`

### フォント・文字

例:

* `--player-font-size-sm`
* `--player-font-size-md`
* `--player-font-size-lg`
* `--player-line-height-base`


### 角丸

* 小 / 中 / 大

### 影

* カード影
* フォーカスリング
* 軽い内側ハイライト（必要なら）

例:

* `--player-shadow-sm`
* `--player-shadow-md`


---

## 2. ハードコード値の整理

`player.css` 内で繰り返し使っている値を、できるだけトークン化してください。

例:

* 同じ色コード
* 同系統の border-radius
* 同系統の padding / gap
* 同系統の box-shadow
* 同系統の font-size

※ 完全にすべてを変数化しなくてよいです。
まずは **繰り返し登場している値** を優先してください。

特に以下は優先して整理してください。

* `#175f9f`
* `#0f4c81`
* `#bcc9d8`
* `#d7dee8`
* `#5f6b7a`
* `#526071`
* border-radius の 8 / 10 / 12 / 14 / 18 / 20
* min-height の 32 / 34 / 46 / 52 / 60
* gap / padding の 8 / 10 / 12 / 14 / 16 / 18 / 20 / 24

全部を完全に統一しなくてよいですが、**意味のある単位に寄せて再利用可能にすること** を重視してください。

---


## 3. トークンの命名(コンポーネント単位で見直しやすい形にする)

命名は、用途が分かるものにしてください。
以下の各ブロックで、トークンが使われるようにしてください。

* ページ全体
* 共通パネル
* メッセージ
* フォーム
* ボタン
* detail
* 年ピッカー
* レスポンシブ

例:

* `--player-color-bg`
* `--player-color-surface`
* `--player-color-border`
* `--player-color-accent`
* `--player-radius-md`
* `--player-space-md`
* `--player-height-control`
* `--player-shadow-card`

細かすぎる命名にしすぎず、今後育てやすい粒度にしてください。

### 補足：トークンの命名方針

命名は、今後も増やしやすいように整理してください。

### 例

* 色: `--player-color-*` でもよい
* 余白: `--player-space-*`
* 半径: `--player-radius-*`
* 影: `--player-shadow-*`

既存の `--player-bg` のような命名を維持してもよいですが、
**命名方針をファイル内で揃える** ことを優先してください。

---

## 4. 既存セクションの置換

すでに整理されている以下のセクション構造は維持してよいです。

* Common base
* Page layout
* Message
* Form
* Button
* Detail
* Admission year picker
* Responsive

その中の具体値を、可能な範囲でトークン参照へ置き換えてください。

---

## 5. 見た目の安定性

トークン整理によって、以下が崩れないようにしてください。

* register / edit / detail の余白感
* ボタンの見た目差（primary / secondary）
* 年ピッカーの見た目
* スマホ幅での収まり
* detail カードの読みやすさ

---

## 6. 今回やらないこと

* 大幅な配色変更
* UIデザインの全面刷新
* パワプロ完全再現
* HTML構造の大規模変更
* JSロジック変更
* コンポーネント分割
* 年ピッカー機能変更

# デザイン方針

以前まとめた「安全にパワプロ風に寄せるデザインガイド」に沿って、
**“ゲームUIとして育てやすい抽象化”** を意識してください。

特に今回は、

* 色をまとめる
* 枠や余白のルールをまとめる
* 独自UIとして変えやすくする

ことが主目的です。

---

# 到達目標

以下を満たせば十分です。

* `player.css` の主要な見た目値がトークン経由で管理されている
* 今後「色だけ変える」「角丸だけ変える」「ゲーム風を強める」などがやりやすい
* 既存UIの見た目は大きく壊れていない
* register / edit / detail の一貫性が保たれている
