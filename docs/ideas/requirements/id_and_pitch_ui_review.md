# ID設計および変化球UI案レビュー

## 概要
本メモは、栄冠データベース作成プロジェクトにおける以下2点について、現時点での整理と推奨方針をまとめる。

1. 学校 / 選手 / snapshot の ID設計
2. 変化球UIにおける「2球種重なりカード」案

現状の実装では、すでに `schools`、`player_series`、`players` の親子構造と、snapshot 時点キーによる管理が入り始めている。このため、本レビューでは新規設計案だけでなく、現行実装との整合も踏まえて方針を整理する。

また、ID設計については「内部ID」「管理用部品コード」「表示用複合コード」を分離する方針を採る。内部処理は単純なIDで安定させ、人間向けの識別は見やすい管理コードで補い、必要な時だけ複合コードを生成する。

## 1. ID設計レビュー

### 1-1. 前提
現状の構造は、概ね以下の責務分離に近い。

- `schools`: 高校
- `player_series`: 同一選手の系列
- `players`: 各時点の選手 snapshot 実体
- snapshot 時期: 入学時、1年夏大会後、1年秋大会後などの時点情報

ID設計では「何を識別するIDか」を分けて考えることが重要である。

- DB内部の行を識別するID
- 人間が見て扱う管理コード
- 同一選手系列を識別するコード
- snapshot の時点を識別するキー
- snapshot の表示順

これらを1つの複合IDに押し込むと、主キー責務と表示責務が混ざり、将来の移行や表示変更に弱くなる。

### 1-2. 基本方針
ID / 管理コード設計では、以下の3層を分けて扱う。

#### 内部ID
DB内部の参照、JOIN、保存処理に使う安定したIDである。人間に見せることは主目的にしない。

例:

- `schools.id`
- `player_series.id`
- `players.id`

#### 管理用部品コード
人間が一覧、CSV、検索、デバッグ、会話の中で対象を区別しやすくするためのコードである。内部IDとは別に保持する。

例:

- `schools.school_code`
- `player_series.series_no`
- `players.snapshot_key`
  - 現行実装では互換名として `players.snapshot_label` を使用している

#### 表示用複合コード
管理用部品コードを必要時に連結して生成する表示用コードである。DB主キーにはしない。

例:

- 学校表示コード: `A7K3M9Q2`
- 選手系列表示コード: `A7K3M9Q2-014`
- snapshot 表示コード: `A7K3M9Q2-014-S05`

この分離により、内部処理はシンプルに保ちつつ、人間向けには分かりやすい識別方法を提供できる。

### 1-3. Base31相当管理コードの前提
学校コードなど、人間が見る管理コードについては、見間違い対策をした Base31 相当の短いコードを採用する方針が有力である。

例:

```text
23456789ABCDEFGHJKLMNPQRSTUVWXYZ
```

除外文字:

- `0`, `O`
- `1`, `I`, `L`

この方式の利点:

- 16進数より同じ桁数で多くの値を扱える
- 36進数より見間違いが少ない
- 管理コードとして人間が扱いやすい
- URL、CSV、デバッグ表示に向く

### レビュー補足: 文字集合の正式呼称
上記の文字集合は `0/O/1/I/L` を除外しているため、厳密には 31 文字である。したがって、本レビューでは正式呼称を **Base31相当の見間違い対策コード** に統一する。

`Base32` という語は比較説明では使ってよいが、schema、helper、docs の正式仕様では `safeBase31`、`humanSafeCode` など、厳密な Base32 と誤解されにくい名前を使う。

### 1-4. ID案の比較

| 案 | 概要 | 評価 | 良い点 | 懸念点 | おすすめ度 |
| --- | --- | --- | --- | --- | --- |
| A | 学校ID・選手ID・snapshot順を結合した複合IDをそのまま主IDにする | 保留 | 見れば意味が分かる、デバッグ時に文脈が見える | 主キー責務と表示責務が混ざる、変更や移行に弱い、JOINやURLで長くなる | △ |
| B | 内部主キーは整数、別に学校コード・選手コード・snapshot情報を持つ | 採用推奨 | 保守しやすい、変更に強い、DB/APIが素直、管理コードも見やすい | コード体系を別で設計する必要がある | ◎ |
| C | 全部を Base31 相当のランダムIDで統一する | 試作向き | コードは短く見栄えがよい、外部公開には強い | 人間が意味を読み取りにくい、学校/選手/snapshotの関係が見えにくい | ○ |
| D | 学校だけ Base31 相当コード、選手は学校内連番、snapshot は `key + order` 分離 | 採用推奨 | 人間に分かりやすい、系列管理しやすい、実装との相性が良い | 初期設計を少し丁寧に決める必要がある | ◎ |

### 1-5. 推奨方針
採用推奨は **B + D の組み合わせ** とする。

つまり、以下の構成を基本方針にする。

- DB内部では単純な整数主キーを使う
- 外部表示や管理用には別コードを持つ
- 学校コードは見間違い対策済みの Base31 相当管理コードを採用する
- 選手は `player_series` 単位で管理する
- 選手系列には学校内連番を持たせる
- snapshot は時点キーと表示順を分離する

### 1-6. 推奨構造

#### `schools`
- `schools.id`
  - 内部整数主キー
- `schools.school_code`
  - 外部向け管理コード
  - 例: `A7K3M9Q2`
  - `UNIQUE` を付ける

#### `player_series`
- `player_series.id`
  - 内部整数主キー
- `player_series.school_id`
  - 所属学校
- `player_series.series_no`
  - 学校内連番
  - DBでは整数として保持する
  - `UNIQUE(school_id, series_no)` を付ける

#### `players`
- `players.id`
  - 内部整数主キー
- `players.player_series_id`
  - 系列参照
- `players.snapshot_key`
  - 例: `entrance`, `y1_summer`, `y1_autumn`
  - docs / API 設計上の正式名称は `snapshot_key` に寄せる
  - 現行実装では互換上 `snapshot_label` を使用している
- `snapshot_order`
  - DB列として持つより、共通定数から導出する方針を推奨する
  - 導出専用値であり、`players` テーブルには追加しない前提とする
- `UNIQUE(player_series_id, snapshot_key)`
  - 現行実装では `UNIQUE(player_series_id, snapshot_label)` に相当する

### 1-7. 表示用コード生成案

#### 学校表示コード
生成元は `schools.school_code` とする。

```text
A7K3M9Q2
```

#### 選手系列表示コード
生成元は `schools.school_code + player_series.series_no` とする。

```text
A7K3M9Q2-014
```

`series_no` はゼロ埋めすると一覧で揃いやすい。初期案では3桁固定を基本とし、必要なら4桁へ拡張する。ただし、ゼロ埋めは表示用ルールであり、DBでは整数として保持する。

#### snapshot 表示コード: 短縮型
生成元は `schools.school_code + player_series.series_no + derived snapshot order` とする。

```text
A7K3M9Q2-014-S05
```

良い点:

- 短い
- 一覧やCSVで扱いやすい
- 学校、選手系列、時点の粒度が読み取りやすい

注意点:

- `S05` 単体では意味が弱いため、画面では表示名も併記するのが望ましい

#### snapshot 表示コード: 意味型
生成元は `schools.school_code + player_series.series_no + snapshot_key` とする。

```text
A7K3M9Q2-014-y2_summer
```

良い点:

- 時点の意味が人間に伝わりやすい

注意点:

- 短縮型より長くなる
- CSVや一覧で横幅を使いやすい

#### 推奨表示
内部管理・一覧用途では短縮型を使い、画面表示では短縮型と表示名を併用するのが扱いやすい。

例:

- コード: `A7K3M9Q2-014-S05`
- 表示名: `2年夏大会後`

この方式なら、以下のバランスが取れる。

- 学校単位で見やすい
- 選手系列が学校内で追いやすい
- snapshot の順序が分かりやすい
- DB内部はシンプルなまま維持できる

### レビュー補足: `snapshot_order` は導出値が望ましい
`S05` のような表示コードは便利だが、`snapshot_order` をDB列として自由に保存すると、時点キーとの不整合が起きる可能性がある。

現行実装では、snapshot の正式時点は `backend/constants/playerSnapshots.js` の `SNAPSHOT_TIMELINE` に定義されており、`getSnapshotOrder()` で順序を導出できる。この方向性を維持し、DBには時点キーを保存し、表示順は共通定数から導出するのがよい。

推奨:

- DB保存: 正式名称は `snapshot_key`
- 現行互換: 実装上は当面 `snapshot_label` を `snapshot_key` 相当として扱う
- 表示順: `SNAPSHOT_TIMELINE` から導出
- 表示コード: `school_code + series_no + derived snapshot order` で生成

新規 docs、API 契約、UI 表示の説明では `snapshot_key` を優先し、既存 schema や既存コードを指す場合だけ `snapshot_label` と明記する。

### 1-8. なぜ複合IDをそのまま主キーにしないのか
複合IDをそのまま主キーにすると、以下の問題が出やすい。

- 主キーの責務と表示コードの責務が混ざる
- 一部の設計変更が全体に波及しやすい
- URL / API / JOIN が長くなりやすい
- snapshot 順序の再定義に弱い
- 学校移行やデータ再編時の柔軟性が下がる
- `school_code`、`series_no`、snapshot 表記のルール変更が主キー変更に直結してしまう

そのため、内部主キーと管理コードを分離する方が保守性が高い。

### 1-9. 現行実装との照合
現行 schema では、以下はすでに実装済みである。

- `schools.id`
- `player_series.id`
- `player_series.school_id`
- `players.id`
- `players.player_series_id`
- `players.snapshot_label`
  - 現行互換名。正式設計名は `snapshot_key`
- `UNIQUE(player_series_id, snapshot_label)`
  - 正式設計上は `UNIQUE(player_series_id, snapshot_key)` に相当する
- `player_pitch_types` などの relation 系テーブルが `players.id` に紐づく構造

一方で、以下はまだ未実装である。

- `schools.school_code`
- `player_series.series_no`
- `UNIQUE(school_id, series_no)`
- 管理コード生成ロジック
- 表示コード生成 helper

また、docs の一部は古い `players` 中心の説明が残っているため、ID設計を固定するタイミングで `docs/design/database_design.md` なども更新するとよい。

### 1-10. 既存データへの採番方針
`school_code` と `series_no` を導入する場合、既存データには移行時に値を付与する。

#### 既存 `schools.school_code`
- 既存 school には migration 時に Base31 相当コードを生成して付与する
- `UNIQUE` 制約に衝突した場合は再生成する
- 一度付与した `school_code` は、外部参照やCSVとの整合を守るため原則として自動再生成しない
- 仕様変更で表示形式を変えたい場合は、保存済みコードを書き換えるのではなく表示 helper 側で吸収する

#### 既存 `player_series.series_no`
- 既存 player_series には `school_id` ごとに連番を付与する
- 初期移行では、同一学校内で `player_series.id ASC` の順に `1, 2, 3...` を振る方針が扱いやすい
- 将来、入学年や登録順を重視したい場合も、migration 手順で明示的に並び順を決める
- `UNIQUE(school_id, series_no)` に衝突した場合は、同一学校内の次の空き番号を使う

#### 再採番の扱い
- 通常運用では `school_code` と `series_no` は安定した管理用部品コードとして扱う
- 表示コードが外部に出た後の自動再採番は避ける
- 学校統合、誤登録、手動移行などで再採番が必要な場合は、管理用導線または明示的な migration として扱う

## 2. 変化球UI「2球種重なりカード」案レビュー

### 2-1. 概要
現在、同方向に2球種ある場合は並列表示している。これに対して、以下の案が検討候補として挙がっている。

- 2枚のカードが重なって見える
- 前面カードと背面カードがある
- 反転 / 入替ボタンで前後を切り替える

この案は、閲覧UIでは省スペース化と見た目の良さが期待できる。一方で、編集UIでは誤編集や見落としにつながる可能性がある。

### 2-2. UI案の比較

| 案 | 概要 | 評価 | 良い点 | 懸念点 | おすすめ度 |
| --- | --- | --- | --- | --- | --- |
| A | 現状のように2球種を並列表示する | 採用中 | 一覧性が高い、同時に比較しやすい | 幅を使う、ラベルが密集しやすい | ○ |
| B | 2枚のカードが重なるように見せ、切替で前後を入れ替える | 試作向き | 省スペース、見た目が良い、2球種あることを表現しやすい | 一覧性が落ちる、編集UIでは事故りやすい | △〜○ |
| C | detail側のみ重なりカード、edit側は並列維持 | 有力 | 確認UIでは見栄えと省スペースを両立しやすい | detail と edit で見え方が変わる | ◎ |
| D | 前面カード＋背面カードの端だけ見せる折衷案 | 試作向き | 2球種あることが視覚的に分かる、完全な隠蔽にならない | 実装難度がやや上がる | ○ |

### 2-3. detail側での評価
detail は確認UIであるため、重なりカード案との相性が良い。

理由:

- 一覧性をある程度保ちつつ、省スペース化できる
- 見た目の面白さを取り込みやすい
- 2球種あることを視覚的に表現できる
- 必要なら切り替えて詳細を見る流れが成立しやすい

現行実装では、detail 側の変化球表示は `frontend/js/pages/player_detail.js` の `renderPitchDirection()` と `renderPitchMovementChart()` 周辺に閉じている。そのため、edit 側の入力UIを壊さず、detail の表示だけを試作しやすい。

### 2-4. edit側での評価
edit は確認より編集が主であり、以下が重要である。

- 今どちらを編集しているか
- もう一方に何が入っているか
- どちらにオリ変や変化量が紐づくか
- 保存時にどう反映されるか

このため、片方が隠れる UI は以下の事故につながりやすい。

- 誤編集
- 見落とし
- 保存事故
- オリ変名の紐づけミス

現行実装では、edit/modal 側の変化球編集は `frontend/js/utils/playerRelations.js` の方向別スロットエディタにまとまっている。各方向に最大2スロットを明示表示する構造は、編集UIとして安全性が高い。

### 2-5. 推奨方針
採用推奨は以下とする。

- detail 側限定で重なりカード試作を検討する
- edit 側は現状の並列可視型を維持する

具体案:

- 前面カードに第1球種を表示する
- 背面カードの端や色だけ少し見せる
- 入替ボタンで前後を切り替える
- hover / click で一時展開できる余地を残す
- 完全に情報が隠れないよう、背面カードにも球種名の一部または存在インジケータを出す

この方式なら、以下のバランスが取れる。

- 2球種あることが分かる
- 省スペースになる
- 完全な情報隠蔽にならない
- 編集UIの安全性を維持できる

### レビュー補足: 試作時の注意点
試作する場合は、最初から複雑な状態管理を入れすぎない方がよい。

初期試作では、以下の範囲に絞る。

- 対象は `player_detail` の閲覧表示のみ
- 2球種ある方向だけ重なり表示にする
- 1球種だけの方向は現状表示のまま
- 切替状態は画面内 state のみで保持する
- 保存データ構造は一切変更しない

UIガイドとの接続:

- 本案は「パワプロ完全再現」ではなく、「野球ゲーム風UIとしての再設計」として進める
- スクリーンショットや既存ゲームUIは、配置、情報の優先度、操作フローの参考に留める
- 色、形、影、カードの重なり方を特定UIに寄せすぎず、独自UIとして成立するかを確認する

確認観点:

- 2球種あることが一目で分かるか
- 背面カードの存在が弱すぎないか
- モバイル幅で文字が潰れないか
- オリ変名と元球種名が誤解なく読めるか
- 変化量メーターが前面と背面で混同されないか

## 3. 採用推奨 / 保留 / 試作向きの整理

| 項目 | 判定 | 理由 |
| --- | --- | --- |
| 内部IDと管理コードの分離 | 採用推奨 | 将来の保守性が高い |
| 学校コードに見間違い対策 Base31 相当コードを採用 | 採用推奨 | 見やすく、短く、実運用向き |
| Base31 相当名称への統一 | 採用推奨 | 現候補は31文字のため、厳密な Base32 とは呼ばない |
| 選手系列を `player_series` で管理 | 採用済み / 継続推奨 | 現行実装と整合している |
| 選手系列に学校内連番 `series_no` を持たせる | 採用推奨 | 管理コード生成と人間の追跡に向く |
| snapshot を `key + order` で扱う | 採用推奨 | 並び順と意味の両方を扱える |
| `snapshot_order` をDB列として自由保存する | 保留 | 時点キーとの不整合が起きやすい |
| `snapshot_order` を共通定数から導出する | 採用推奨 | 現行実装と相性が良い |
| 複合IDをそのまま主キーにする | 保留 | 可読性はあるが保守性が落ちやすい |
| 変化球UIを edit/detail 両方で重なりカード化 | 保留 | edit 側で誤操作の懸念がある |
| detail側のみ重なりカード試作 | 試作向き | 見栄え改善と省スペース化のバランスがよい |
| 背面カードの端を見せる折衷案 | 試作向き | 2球種存在が分かりやすい |

## 4. 推奨する次アクション

### 優先度高
1. ID設計を固定する
2. 学校コードの Base31 相当仕様を決定する
3. `school_code` と `series_no` の schema 追加方針を決める
4. 既存 school / player_series への採番手順を決める
5. 重複時や再生成時の扱いを決める
6. snapshot 時点キーの正式名称を `snapshot_key` として固定する
7. `snapshot_order` はDB列ではなく導出値として扱う方針を docs に明記する
8. docs へ設計を反映する

### 優先度中
1. `schools.school_code` の生成 helper を実装する
2. 新規 `player_series` 作成時の次番号採番 helper を実装する
3. 管理コード表示 helper を追加する
4. `player_detail` 側で変化球UI重なりカードを小さく試作する
5. edit側への導入可否は、detail試作後に再評価する

### 優先度低
1. URL/APIで管理コードを使うか、内部IDを使い続けるかを検討する
2. CSV export/import に管理コードを含めるかを検討する
3. OCR導線で管理コードをどこまで扱うかを検討する

## 5. 結論

### ID設計
採用推奨は、複合IDをそのまま主キーにする方式ではなく、以下の構成である。

- 内部主キーは整数ID
- 外部表示用に管理コードを別で持つ
- 学校コードは見間違い対策済みの Base31 相当コード
- 選手は `player_series` で系列管理
- 選手系列は学校内連番で追えるようにする
- snapshot は正式名称 `snapshot_key` として時点キーを保存し、表示順は共通定数から導出する

この構成は、現行実装の `player_series` / `players(snapshot)` 構造と整合しており、将来の移行や表示変更にも強い。

### 変化球UI重なりカード案
detail 側限定なら、前向きに試作してよい案である。

一方で、edit 側の主UIにするには、一覧性と誤操作防止の観点から慎重に扱うべきである。現時点では、edit 側は方向別スロットを明示表示する現行方式を維持するのが安全である。
