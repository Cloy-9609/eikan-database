## NDL OCR 検討メモ（草案）

### 概要
国立国会図書館系の OCR 公開物（NDLOCR / NDLOCR-Lite）は、日本語 OCR の候補として有力である。
本プロジェクトでは、選手画像・能力画面・各種テキスト情報の読み取り補助を将来的に実装する可能性があるため、候補技術の一つとして記録しておく。

---

### 1. NDL OCR を利用するメリット・目的

#### メリット
- 日本語 OCR として検討価値が高い
- 公開ライセンス上、将来的な商用利用を検討しやすい
- OSS 系公開物のため、ブラックボックス API 依存を避けやすい
- ローカル実行構成を取れれば、外部 API 使用料や送信コストを抑えられる
- 将来、OCR 精度改善のための前処理や後処理を自前で設計しやすい

#### 目的
- 手入力負荷の軽減
- 選手登録時の入力補助
- OCR 結果を候補値として提示し、ユーザーが確認・修正できる運用の実現
- 将来的な一括登録・半自動登録の基盤作成

---

### 2. NDL OCR で向上する機能

#### 向上が期待できる機能
- 選手画像からの文字情報抽出
- 特殊能力名・選手名・学校名などの文字認識補助
- 入力フォームへの仮入力
- OCR 結果を JSON 化し、登録候補として保持する機能
- 将来的な「画像アップロード → 候補抽出 → 確認して保存」の流れ

#### 特に相性が良さそうな用途
- 日本語テキストの候補抽出
- 手入力だと負担が大きい項目の補助
- 数値以外のラベル認識補助

#### 注意
- ゲーム UI 画像は一般的な文書 OCR の想定対象と完全一致しない可能性がある
- 小さい数字、装飾文字、色付き UI、背景付き文字では精度検証が必要
- そのため、OCR 結果は「正解」として直接保存するより、「候補値」として扱う設計が安全

---

### 3. NDL OCR を利用する場合の構築設計

#### A. ローカル OCR エンジンとして利用する構成

##### イメージ
- ユーザーが画像をアップロード
- バックエンドまたは補助スクリプトで OCR 実行
- OCR 結果を JSON として返却
- フロントで候補表示し、ユーザー確認後に保存

##### 構成案
- Node.js / Express の既存バックエンドとは別に、OCR 用の Python 実行環境を用意
- Node 側から Python スクリプトを呼び出す
- 画像前処理（切り抜き、拡大、二値化など）を追加可能にする
- OCR 出力をそのまま DB 保存せず、一旦候補データとして画面表示する

##### 向いているケース
- 開発初期の検証
- 個人利用・ローカル利用中心
- サーバー費用を増やしたくない場合
- OCR の精度検証を段階的に進めたい場合

---

#### B. サーバー組み込み型として利用する構成

##### イメージ
- Web アプリ上で画像アップロード
- サーバー側で OCR 実行
- API 経由で結果返却
- フロントで確認画面を表示して保存

##### 構成案
- サーバーに OCR 実行環境を同居
- 画像アップロード API と OCR 実行 API を分離
- OCR 実行ジョブと DB 保存処理を分離
- 将来的にはキュー処理や非同期化も検討

##### 向いているケース
- 複数端末から同じ環境で使いたい場合
- ローカル環境差分を減らしたい場合
- Web アプリとして一体運用したい場合
- 将来的にユーザー数増加を見込む場合

---

### 4. 各構築設計についての懸念

#### A. ローカル OCR エンジン構成の懸念
- Node.js 構成とは別に Python 環境管理が必要
- 開発 PC ごとの差異が出やすい
- 配布時にセットアップ手順が増える
- 将来的に Web 公開する際は構成を再設計する可能性が高い
- OCR 処理が重い場合、ローカルマシン性能に依存する

#### B. サーバー組み込み構成の懸念
- サーバー負荷が増える
- OCR 実行時間が長いとレスポンス設計を工夫する必要がある
- デプロイ先で Python / モデル / 依存関係を安定稼働させる必要がある
- 将来的な同時実行数増加に備えた設計が必要
- 画像アップロードを伴うため、保存ポリシー・容量管理・セキュリティ設計が必要

#### 共通懸念
- ゲーム画面向け OCR として十分な精度が出るかは未検証
- 数値認識と文字認識で別手法を併用した方が精度が高い可能性がある
- OCR の誤認識を前提とした UI / 確認フローが必要
- 導入前にサンプル画像での精度比較が必要

---

### 5. 商用利用についての是非

#### 現時点の認識
- 公開ライセンス上は、将来的な商用利用を検討しやすい候補と考えられる
- ただし、実際に採用する場合は以下を再確認すること
  - 対象リポジトリの README
  - LICENSE
  - 依存ライブラリのライセンス
  - 利用するモデルや重みファイルの条件
  - 配布形態（SaaS / ローカル配布 / デスクトップ同梱）の違い

#### このプロジェクト目線での判断
- 「候補技術」としてメモしておく価値は高い
- ただし、商用化判断は導入時ではなく、実際に公開・収益化・配布形態が見えた段階で最終確認する
- そのため、現段階では「商用利用の可能性あり。ただし採用時に再確認必須」と記載するのが妥当

---

### 6. 組み込むとしたらどの phase になりうるか

#### 想定
- Phase 1:
  - 非導入
  - まずは手動入力・基本 CRUD・画面設計を優先
- Phase 2:
  - OCR 検証フェーズ
  - サンプル画像を用いた読み取り実験
  - どの項目に OCR を使うか切り分け
- Phase 3:
  - OCR の実運用導入候補
  - 画像アップロード → 候補抽出 → 確認保存の流れを実装
- Phase 4 以降:
  - 精度改善
  - 前処理追加
  - 数値認識専用ロジックやテンプレートマッチングとの併用検討

#### 現時点の結論
本プロジェクトでは、NDL OCR は「今すぐ必須の基盤」ではなく、
**Phase 2 以降で検証し、実装候補として判断する技術**
として扱うのが妥当。

---

### 7. 現段階での暫定方針
- docs 上では「採用決定」ではなく「検討候補」として記録する
- OCR は直接保存ではなく、候補提示型を前提とする
- 実装判断前に、最低限以下を確認する
  1. ライセンス再確認
  2. サンプル画像での精度検証
  3. Node 既存構成との接続方法確認
  4. ローカル構成とサーバー構成のどちらを先に試すか決定
