# UI 見た目ルールガイド

## 1. 目的
本プロジェクトの UI は、既存画面を壊さずに少しずつ整える方針を取る。
ここでいう「整える」は、派手な再デザインではなく、余白・見出し・色・ボタン役割・フォーム部品の統一感を高めることを指す。

今後 Codex や開発者が UI を触るときは、まずこのガイドを基準に判断する。

## 2. このプロジェクトの UI 方針
- 既存 UI を壊さずに整えることを最優先にする。
- 画面ごとの自由な見た目変更より、共通ルールへの寄せを優先する。
- 視認性、操作の分かりやすさ、情報の優先順位を、装飾より優先する。
- 学校画面と選手画面で使っている基本部品を、今後の基準 UI として扱う。
- 新しい装飾を足すより、既存の余白、色、角丸、枠線、見出し階層を揃える。

## 3. 現在の基準ファイル
- 学校画面のパネル、フォーム、検索、テーブル: `frontend/css/schools.css`
- 選手画面のページトーン、フォーム、ボタン、詳細カード: `frontend/css/player.css`
- 西暦ピッカー共通部品: `frontend/css/components/admission_year_picker.css`
- 西暦ピッカー共通ロジック: `frontend/js/components/admissionYearPicker.js`

新規 UI を追加するときは、まず上記の既存部品に寄せられないか確認する。

## 4. パーツごとの基本ルール

### 4.1 パネル
- 基本は白、またはごく薄い青系背景を使う。
- 枠線は 1px の薄い青灰系を基本とする。
- 角丸は 16px から 20px 程度を基準とし、急に角張らせない。
- 内側余白は 20px 前後を基準にし、モバイルでは少し詰める。
- 見出しと本文の間は 16px 前後を基準とする。
- 濃い影や重い立体感は避け、薄いシャドウで十分とする。

### 4.2 見出し
- ページタイトルが最上位で、最も強いサイズと太さを持つ。
- パネル見出しはページタイトルより一段弱くする。
- 補助説明文は見出しより明確に弱い色とサイズで見せる。
- 補助説明文は長くしすぎず、1 から 2 文程度に収める。
- 見出し階層を増やしすぎず、ページタイトル、パネル見出し、補助説明の 3 段で整理する。

### 4.3 フォーム入力欄
- `input` / `select` / `textarea` は既存形を基準部品として扱う。
- 高さは 44px から 46px 前後、角丸は 10px から 12px 前後を維持する。
- 枠線は薄い青灰系の 1px を基本とする。
- 背景は白を基本とし、濃い塗りや強いグラデーションは入れない。
- `padding` は現状に近い 10px から 16px 程度を保ち、極端に詰めたり広げたりしない。
- フォーカス時は青系の枠線と薄いリングで見せる。強い発光表現にはしない。
- `textarea` は必要に応じて縦方向リサイズ可とし、全面再設計はしない。
- 補助文はグレー系で短く保ち、説明を詰め込みすぎない。

### 4.4 ボタン
- 主ボタンは青系塗りを使い、主要操作だけに使う。
- 副ボタンは白背景 + 枠線で、戻る、リセット、補助操作に使う。
- 危険ボタンは赤系に限定し、削除など破壊的操作だけに使う。
- 高さは 44px から 46px 前後、角丸は 10px から 12px 前後を基準とする。
- hover は色差と軽い浮き上がり程度に留める。
- disabled は押せないことが明確に分かる程度に彩度とコントラストを落とす。
- 役割ごとの色分けは崩さない。主ボタンの青、副ボタンの白枠、危険ボタンの赤を混同しない。

### 4.5 テーブル / 一覧
- ヘッダー行はごく薄い背景色で本文と区別する。
- 本文行の余白は 10px から 12px 程度、行高は 1.5 前後を基準とする。
- リンクは青系で見せ、hover 時だけ下線や色変化を加える。
- hover は必要な場合のみ薄い背景変化に留め、派手な演出は避ける。
- 列ごとの情報密度を上げすぎず、読みやすさを優先する。

### 4.6 メッセージ
- success は緑系の薄い背景 + 緑系テキストで見せる。
- error は赤系の薄い背景 + 赤系テキストで見せる。
- empty は中立的な薄い面と落ち着いた本文色で見せる。
- 強い警告色や点滅は使わない。

## 5. 西暦ピッカーの扱い
西暦ピッカーは、本プロジェクトの基準 UI パーツの 1 つとして扱う。
この部品はすでにかなり仕上がっている既存コンポーネントであり、今後の UI 整理でも不用意に形を変えない。

### 5.1 基本方針
- 共通ロジックは `admissionYearPicker.js` を継続利用する。
- default / compact variant の役割分担は維持する。
- digits の中央寄せ、数値の視認性、上下ボタンの縦構成、選択中ラベル、エラー表示の考え方を維持する。
- 年範囲、hidden input 更新、桁ごとの操作ロジックは見た目調整の都合で変えない。

### 5.2 variant の使い分け
- default variant は選手画面系の標準サイズとして扱う。
- compact variant は学校画面系の省スペース表示として扱う。
- compact は小さくしてよいが、押しづらさや読みづらさが出るほど縮めない。
- 見た目の調整が必要な場合は、まず variant 内の寸法調整で吸収する。

### 5.3 禁止に近い扱い
- DOM 構造を別物に変えない。
- 4 桁構成を崩さない。
- 上ボタン / 数字 / 下ボタンの配置を変えない。
- 突然フラットな別 UI や別世界の色に置き換えない。
- 年ピッカーだけ極端に濃い影や強いアニメーションを足さない。

### 5.4 今後の微調整ルール
- 微調整は `default` または `compact` の中で行う。
- レイアウト不具合は、まず配置や親要素幅の問題として切り分ける。
- 部品自体の再設計は最後の手段とする。
- 「極力崩さない」を常に優先する。

## 6. 余白ルール
余白は種類を増やしすぎず、まず次の 3 段階を基準に使う。

- 小: 8px
- 中: 16px
- 大: 24px

補足:
- 既存コンポーネント内ですでに使っている 6px、10px、12px、18px などは、その部品の微調整値として維持してよい。
- ただし新規 UI を作るときは、まず 8 / 16 / 24 を基準に考え、必要があるときだけ局所的に外す。
- 余白トークンを増やしすぎない。

## 7. 色の役割ルール
色は好みではなく役割で使う。

- 主操作: 青系
- 副操作: 白 + 枠線
- 危険操作: 赤系
- 補助説明: グレー系
- 成功: 緑系
- エラー: 赤系
- パネル背景: 白、またはごく薄い青系

補足:
- 新しいアクセント色を増やしすぎない。
- 役割の違う操作に同じ強さの色を使わない。
- グラデーションは既存部品の範囲内で使い、全面的に強くしない。

## 8. 今後の UI 修正時の禁止事項 / 注意事項

### 8.1 禁止事項
- 年ピッカーの構造をむやみに変えない。
- 既存 `input` / `select` / `textarea` の形をむやみに変えない。
- 新しい画面だけ別世界のデザインにしない。
- 色を増やしすぎない。
- 余白サイズを増やしすぎない。
- ボタン役割の色分けを崩さない。
- 影やグラデーションを強くしすぎない。

### 8.2 注意事項
- まず既存 CSS と既存画面を読み、すでにある部品に寄せる。
- 共通コンポーネント化できるものは、ページ個別の応急処置より共通化を優先する。
- ただしレイアウト不具合の修正は、問題画面に限定した局所補正でもよい。
- 見た目の違和感があるときは、サイズより先に配置、余白、親幅、行間を確認する。
- 見た目に違和感がある場合は、まず「配置」「余白」「幅」「高さ」「色」の順で原因を切り分け、装飾追加は最後に検討する。
- 新しい装飾を足すより、不要な差異を減らす。
- UI を修正する場合は、変更前後で対象画面の実ブラウザ確認を行い、必要に応じてスクリーンショット比較で採用判断する。

## 9. 今回あえて実装しない範囲
今回のガイドは、次のような大きな変更を前提にしない。

- 大きな配色刷新
- アイコン体系の全面導入
- 強いアニメーション
- school_detail の操作メニュー刷新
- player_detail / player_edit の全面再レイアウト
- 年ピッカーや入力欄の全面再設計

## 10. 運用メモ
- UI を直すときは、まず「既存の基準部品に寄せられるか」を確認する。
- 学校画面と選手画面で同じ役割を持つ UI 部品は、可能な限り同じ見た目ルールへ寄せ、画面ごとの偶発的な差異を増やさない。
- 学校画面と選手画面で差がある場合は、役割差か偶発的なズレかを先に切り分ける。
- 判断に迷った場合は、新しい見せ方を足すより既存 UI を維持する方を優先する。
