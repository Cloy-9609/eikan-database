# サーバーの起動方法（Phase1 時点）
1. ターミナルにて `npm run dev` を入力
2. ブラウザで `http://localhost:3000/pages` または `http://localhost:3000/pages/schools.html` に入る。
3. 取り敢えず F12 にてコンソールを開き、問題の有無を確認する。

補足:
- backend 変更時の自動再起動を使う場合は `npm run dev` だけを起動する。
- `node backend/app.js` と `npm run dev` を同時に起動すると、同じ port を取り合うため避ける。
- watch なしで単純起動したい場合のみ `npm start` を使う。

## 修正プロンプトをするときの有効なテンプレ
### 確認について
- 今回はローカルサーバー起動や HTTP 200 確認までは不要です
- コード差分、構造説明、影響範囲の整理、必要なら構文チェックのみ行ってください
- 実ブラウザでの最終見た目確認はユーザー側で行います
- そのため、確認結果には
  - どのCSS/JS/HTMLをどう変えたか
  - どの既存仕様を維持したか
  - 実ブラウザで確認すべきポイント
  を記載してください

## プロンプト作成を指示するときに有効なテンプレ
---
### その一
作成するプロンプトが長大・複雑になる場合は、一度にすべてを出力しようとせず、実装内容を適切な単位に分割してください。

まず、全体をどのようなタスクに分割するか、各タスクの実施順序と依存関係を整理してください。その後、分割したタスクごとにCodexへ渡すプロンプトを1つずつ順番に作成してください。

原則として、1つのプロンプトは「1回の実装・確認・PRで完結しやすい範囲」とし、過度に細かく分割しすぎないようにしてください。また、各プロンプトは単独でも実行内容を理解できるようにしつつ、前のタスクで行った変更や前提条件がある場合は明記してください。

一度の回答では、最初に全体の分割案を示したうえで、第1段階のプロンプトまでを作成してください。以降は、私から続行を指示した際に、次の段階のプロンプトを順に作成してください。

各プロンプトには、従来どおり以下を含めてください。

・タスクの目的
・実装内容と対象範囲
・変更対象となる可能性があるファイル
・具体的な実装指示
・既存機能への影響と注意点
・確認方法、テスト項目
・完了条件
・優先度
・リスク
・規模感
・Codexによる自動マージの可否判断

複数のプロンプト間で、指示の重複、実装漏れ、仕様の不整合が起こらないように管理してください。
---
### その二
この依頼に対するCodex用プロンプトが長大になる場合は、一度にすべてを作成せず、実装・確認・PRが行いやすい単位に分割してください。

最初に全体の分割案、実施順序、依存関係を示し、その後、第1段階のプロンプトのみを作成してください。以降は、私が続行を指示するたびに、次のプロンプトを順番に作成してください。

各プロンプトは単独で実行可能な内容とし、前段階の変更を前提とする場合は、その前提を明記してください。また、優先度、リスク、規模感、自動マージ可否も併記してください。

### よく使うコマンド
- `git remote prune origin`  
・頻繁にクラウド上で更新がかかるため、手動でリモートブランチを更新する必要があるときに使用する。

- 
`
  git archive \
    --format=zip \
    --output="../eikan-database-source-$(date +%Y%m%d-%H%M).zip" \
    HEAD
`
・node_moduleを含まない、軽量なバックアップのzipファイルを作成するときに使用する。
